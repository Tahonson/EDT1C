# EDT + MCP + IDE (Cursor / VS Code)

Практическое руководство по разработке в 1С через EDT с поддержкой ИИ (Напарник, MCP, Cursor/VS Code), с акцентом на безопасный рабочий процесс, Git и типовые кейсы команды.

Источник материала и база best practices: [bulzavr.ru / Разработка EDT](https://bulzavr.ru/logseq/#/page/it%2F1%D1%81%2F%D1%80%D0%B0%D0%B7%D1%80%D0%B0%D0%B1%D0%BE%D1%82%D0%BA%D0%B0%20c%20edt%20%D0%B8%20mcp%2F%D1%80%D0%B0%D0%B7%D1%80%D0%B0%D0%B1%D0%BE%D1%82%D0%BA%D0%B0%20edt).

## 1) EDT + Напарник

### 1.1 Требования и развертывание

Алгоритм установки рабочей среды:

1. Шаг 1. Платформа 1С. Скачайте с `releases.1c.ru` и установите актуальную платформу (рекомендуется `8.3.24` и выше).
2. Шаг 2. EDT. Скачайте с `releases.1c.ru` версию EDT не ниже `2026.1` (содержит актуальные интеграции с AI).
3. Шаг 3. Java. Установите `BellSoft Liberica JDK` (версия — согласно требованиям скачанной EDT).
4. Шаг 4. Плагин Напарника. В EDT откройте `Help -> Install New Software`, добавьте репозиторий плагина Напарника и установите плагин.
5. Шаг 5. Авторизация. Перезапустите EDT, затем в появившемся окне выполните авторизацию под УЗ портала ИТС.

Алгоритм генерации:

1. Шаг 1. Установите курсор в пустую строку внутри процедуры.
2. Шаг 2. Напишите комментарий-инструкцию (промпт).
Пример хорошего промпта:

```bsl
// Сформируй запрос к регистру накопления ОстаткиТоваров. Обороты, условие по Складу (из параметров), сгруппируй по Номенклатуре.
```
Пример плохого промпта:

```bsl
// Получи остатки
```
3. Шаг 3. Нажмите `Enter`. Напарник сгенерирует текст ниже.
4. Шаг 4. Обязательная проверка: убедитесь, что имена реквизитов/переменных совпадают с реальными метаданными.

### 1.4 Правила комментирования

1. Обычные комментарии (`//`): используйте для постановки задач Напарнику внутри алгоритма.
2. Документирующие комментарии: строго перед объявлением метода.

Алгоритм:

1. Шаг 1. Введите `//` перед процедурой и нажмите `Enter`. EDT создаст шаблон.
2. Шаг 2. Заполните типы и назначение параметров. Это критично для контекста AI в других модулях.

Пример:

```bsl
// Получает артикул по ссылке на номенклатуру.
// Параметры:
//  Товар - СправочникСсылка.Номенклатура - Искомый элемент.
// Возвращаемое значение:
//  Строка - Артикул товара.
```

## 2) EDT + MCP

### 2.1 Настройка MCP для 1С (для работы в Cursor)

Алгоритм подключения:

1. Убедитесь, что MCP-сервер для 1С доступен в вашей среде.
2. В Cursor откройте настройки: `Cursor Settings -> Features -> MCP`.
3. Добавьте MCP-сервер и проверьте, что есть нужные вам права (read/write там, где это разрешено).
4. Протестируйте 2–3 реальных сценария: поиск по коду, разбор структуры проекта, правки и git-операции (в рамках разрешений).

**Скриншот (замените): настройки MCP в Cursor:**

![Скриншот: настройки MCP (замените)](docs/screens/shot-cursor-mcp-settings.png)

Шаблон `mcp.json`:

```json
{ "mcpServers": { "1c-docs": { "url": "https://bulzavr.ru/mcp" }, "EDT MCP Server": { "type": "sse", "url": "http://localhost:8765/mcp" } } }
```

На практике самый удобный сценарий — работать через Cursor/VS Code: там проще держать контекст файлов и видеть diff рядом с изменениями.

## 3) EDT + IDE (Cursor / VS Code)

### 3.1 Выбор решения / модели
Настройки моделей (Cursor -> Settings -> Models):

1. `Primary model` (для сложных задач): `Claude 3.7 Sonnet` — для архитектуры и сложных запросов.
2. `Fast model` (для автокомплита и мелких правок): `Claude 3.5 Haiku` — для быстрых итераций.

Для критичных изменений всегда делайте ручную проверку и прогон сценариев.

Алгоритм настройки рабочей области:

1. Установите плагин подсветки 1С в Cursor (например: `1C (BSL) Language Support` или аналогичный).
2. Создайте файл `.cursorrules` в корне проекта.
3. Вставьте базовые правила (пример):

```text
Язык программирования: 1С (BSL).
Использовать только русский синтаксис.

Политика изменений:
- Менять только то, что требуется задачей (без “скрытых” архитектурных рефакторингов).
- Перед любыми изменениями, влияющими на метаданные, сверять имена реквизитов/методов по текущему проекту (через EDT).
- По возможности расширять функциональность через расширения, а не через правку “ядра”.
```

### 3.3 Методология промптов в Cursor/IDE
Алгоритм постановки задачи (Ctrl+L / Cmd+L):

1. Дайте контекст через `@`.
Пример: `@Документ.Реализация.МодульОбъекта.bsl @ОбщийМодуль.Скидки.bsl`
2. Опишите задачу структурно.
Пример: В прикрепленном модуле документа найди процедуру `ОбработкаПроведения`. Добавь вызов функции `РассчитатьСкидку` из модуля `Скидки` перед записью движений.
3. Вложения: если документ/форма незнакомы — добавьте через `@` XML файл структуры (например `Document.Реализация.xml`), чтобы Cursor “увидел” реквизиты.
4. Критерии готовности: что должно получиться и как проверить.

### 3.4 Фичи Cursor/IDE по работе с кодом

Полезные возможности:

- inline-редактирование и точечная генерация;
- режим агента для многошаговых задач;
- работа с diff в процессе генерации;
- быстрый переход по проекту и поиск контекста;
- интеграция с git-процессами (коммит, PR-сценарии, ревью).

Рекомендация по горячим клавишам:

- Зафиксируйте в команде единый "минимум" шорткатов: вызов агента, принятие/отмена автодополнения, переход к diff, запуск поиска.
- Добавьте внутренний cheatsheet под вашу ОС (Windows/macOS/Linux), так как раскладки отличаются.

## 4) Кейсы

### 4.1 Доработка основной конфигурации (типовой код)

Алгоритм:

1. В EDT найдите нужный общий модуль или модуль объекта.
2. Выделите участок кода и поставьте задачу для Напарника/ Cursor.
3. Промпт (пример): Внедри проверку переменной `Контрагент` на заполненность. Если пустая - вызови `ВызватьИсключение` с текстом "Не заполнен контрагент".
4. После генерации примените результат (Apply/Tab) и сохраните файл.
5. Перейдите в EDT и обновите кэш/ИБ: F5 (кэш) -> F7 (обновить ИБ).

### 4.3 Разработка внешнего отчета (на СКД)

Алгоритм:

1. В EDT или Конфигураторе создайте внешний отчет, добавьте макет СКД.
2. Откройте Cursor и сформируйте промпт на текст запроса для СКД 1С.
3. Промпт (пример): Напиши текст запроса для СКД 1С. Нужно соединить ВТ `ОстаткиТоваров` с таблицей `Справочник.Номенклатура` (Номенклатура/Склад). Выведи `Артикул`, `Наименование` и `КоличествоОстаток`.
4. Скопируйте запрос в конструктор СКД в EDT/Конфигураторе.
5. Настройку полей и группировок выполняйте визуально в EDT, затем проверьте отчет на реальных данных.

2. Откройте Cursor, выделите проблемный блок кода.
3. В Chat вставьте: Код падает с ошибкой — вставьте скопированный текст ошибки из 1С. Исправь.
4. Примените исправление (Apply) и повторите шаги воспроизведения.

Алгоритм инициализации:

1. Выгрузить конфигурацию в проект EDT (папка `src`).
2. Открыть терминал в корневой папке проекта.
3. Выполнить команды:
   - `git init`
   - создать `.gitignore` (исключите служебные файлы EDT/временные каталоги согласно вашему стандарту)
   - `git add .`
   - `git commit -m "init: Базовая выгрузка конфигурации"`
4. Подключить remote и отправить основную ветку (пример: `master`/`main`):
   - `git remote add origin URL_репозитория` (замените на ваш URL репозитория)
   - `git push -u origin master` (или `main`)

### 5.2 Подключение разработчика к проекту с Git

Алгоритм начала работы:

1. Клонировать репозиторий (замените URL на ваш):
   - `git clone URL_репозитория` (замените на ваш URL репозитория)
2. Импортировать проект в EDT:
   - `Файл -> Импорт -> General -> Проекты из папки или архива`
3. Привязать информационную базу для запуска (ПКМ по проекту -> `Свойства -> 1С:Предприятие`):
   - создать пустую ИБ и привязать её
4. Выполнить `F7` и убедиться, что сборка/обновление проходит.

### 5.3 Инструкция разработчика по Git (ежедневный цикл)

Алгоритм выполнения задачи:

1. Создать feature-ветку:
   - `git checkout -b feature/TASK-ID-КраткоеОписание`
2. Пишите код в EDT/Cursor.
3. Подготовка к коммиту (синхронизация/сверка):
   - в EDT откройте Git Staging/Синхронизация
   - внимательно проверьте измененные файлы и отсутствие случайных массовых переформатирований метаданных

**Скриншот (замените): Git Staging/Синхронизация в EDT:**

![Скриншот: Git Staging (замените)](docs/screens/shot-git-staging.png)

4. Коммит:
   - `git add .`
   - `git commit -m "feat: Реализована логика по задаче TASK-ID"`
5. Отправка:
   - `git push origin feature/TASK-ID-КраткоеОписание`
6. Merge Request/PR:
   - создайте PR/MR и дождитесь ревью/правок

### 5.4 Взаимодействие Git и EDT (Выгрузка/Загрузка)

При получении изменений (`git pull`):

1. Выполните `git pull` в терминале.
2. В EDT дождитесь перестроения рабочей области (фоновый процесс).
3. Выполните `F7` (применение изменений к локальной базе 1С).

При смене веток (`git checkout`):

1. Смените ветку (в Cursor/терминале).
2. В EDT на корне проекта выполните `F5` и дождитесь перестроения.
3. Выполните `F7`.

При мерджах:

- Не мержьте “вслепую”: обязательно просматривайте diff и понимайте влияние на бизнес-сценарии.
- После merge прогоните воспроизводимые smoke/regression-тесты.

Самый критичный процесс. Строго соблюдайте последовательность.

Исходные данные: у вас есть ветка `vendor` (чистая конфигурация поставщика) и ветка `master/develop` (ваша доработка).

Алгоритм обновления:

1. Обновление ветки `vendor`:
   - `git checkout vendor`
   - обновите базу в EDT (F5)
   - в EDT: ПКМ на проекте -> `Импорт` -> `1С:Предприятие` -> `Конфигурация из файла`
   - выберите новый `.cf` от 1С и дождитесь окончания разбора
   - `git add .` и `git commit -m "vendor: Обновление типовой конфигурации до релиза X.X.X.X"`
2. Подготовка к слиянию:
   - `git checkout master` (или вашу ветку разработки)
   - в EDT дождитесь перестроения проекта на старую ветку
3. Запуск слияния:
   - в терминале: `git merge vendor`
4. Разрешение конфликтов (Merge Tool):
   - если конфликтов нет — переходите к шагу 6
   - если конфликты есть — в EDT откройте Merge Tool по конфликтному метаданному объекту
   - перенесите нужные блоки кода и реквизиты в центральную панель, сохраните
   - в терминале завершите merge после разрешения конфликтов (при необходимости отдельным commit)

6. Обновление базы:
   - в EDT выполните `F7` и проведите стандартное обновление базы данных 1С (при необходимости через F5/запуск предприятия для обработчиков обновления).

## Рекомендуемый командный стандарт (кратко)

- Планировать нетривиальные задачи до генерации кода.
- Давать ИИ ограниченный, но достаточный контекст.
- Работать малыми изменениями и регулярно ревьюить diff.
- Не отключать человеческий контроль: тесты и код-ревью обязательны.
- Поддерживать единые версии инструментов и единый git-процесс по команде.
