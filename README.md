# EDT + MCP + IDE (Cursor / VS Code)

Практическое руководство по разработке в 1С через EDT с поддержкой ИИ (Напарник, MCP, Cursor/VS Code), с акцентом на безопасный рабочий процесс, Git и типовые кейсы команды.

Источник материала и база best practices: [bulzavr.ru / Разработка EDT](https://bulzavr.ru/logseq/#/page/it%2F1%D1%81%2F%D1%80%D0%B0%D0%B7%D1%80%D0%B0%D0%B1%D0%BE%D1%82%D0%BA%D0%B0%20c%20edt%20%D0%B8%20mcp%2F%D1%80%D0%B0%D0%B7%D1%80%D0%B0%D0%B1%D0%BE%D1%82%D0%BA%D0%B0%20edt).

## 1) EDT + Напарник

### 1.1 Версии платформы, развертывание проекта, версии EDT и нюансы

Минимальный базовый чек-лист:

1. Убедиться, что версии `1С:Предприятие`, `EDT`, Java и плагинов совместимы между собой.
2. Поднять/подключить информационную базу нужной версии.
3. Создать или открыть workspace в `EDT`.
4. Импортировать конфигурацию (и расширения, если используются).
5. Проверить сборку проекта и отсутствие ошибок метаданных.
6. Настроить Git (ignore, ветки, права доступа, удаленный origin).
7. Зафиксировать "эталонное" состояние проекта первым корректным коммитом.

Практические рекомендации:

- Держите единый стандарт версий в команде (платформа/EDT), иначе получите ложные diff и проблемы со сборкой.
- Никогда не смешивайте "экспериментальные" обновления EDT с продуктивными ветками без отдельной проверки.
- Перед началом задачи подтверждайте, что база и исходники синхронизированы.

### 1.2 Как работает автоматическое продолжение кода

Автодополнение в IDE работает по текущему контексту:

- активный файл и позиция курсора;
- соседние символы, сигнатуры, паттерны проекта;
- правила/инструкции, если они настроены;
- иногда подключенный контекст репозитория.

Что важно:

- Чем точнее локальный контекст (названия, стиль, сигнатуры), тем качественнее продолжение.
- Автопродолжение хорошо для "локальных" участков кода; архитектурные изменения лучше делать через явный промпт и план.
- Принято сначала получить предложение, потом вручную проверить корректность по бизнес-логике.

### 1.3 Как правильно делать генерацию кода и какие ограничения

Рекомендованный подход:

1. Сначала короткий план задачи (что меняем, где, зачем).
2. Затем ограничить область генерации: конкретный модуль/процедура/файл.
3. Попросить код "в стиле текущего проекта" и без лишних рефакторингов.
4. После генерации сразу прогнать локальную проверку и ревью diff.

Ограничения генерации:

- Модель может синтаксически сгенерировать корректный код, но не угадать бизнес-контракт.
- Возможны "галлюцинации" API/методов, если контекст неполный.
- На больших промптах падает точность: шум в контексте ухудшает результат.
- Нельзя пропускать ручную проверку и тестирование.

### 1.4 Комментарии к процедурам/функциям/модулю. Документирующий комментарий

Обычный комментарий:

- Поясняет "почему так сделано" в сложном месте.
- Короткий, предметный, привязан к конкретной логике.

Документирующий комментарий:

- Описывает контракт API: назначение, параметры, возвращаемые значения, исключения, ограничения использования.
- Должен быть стабильным и поддерживаться при изменении сигнатуры.

Рекомендации:

- Не дублировать код словами.
- Писать комментарии там, где без них сложно понять решение.
- Для публичных/переиспользуемых процедур предпочитать документирующий стиль.

## 2) EDT + MCP

### 2.1 Настройка MCP EDT

Что дает MCP в связке с EDT:

- доступ ИИ к инструментам проекта (поиск, анализ, git-процессы, автоматизация действий);
- более точную работу по репозиторию относительно "чистого чата".

Базовая схема настройки:

1. Установить/подключить MCP-сервер(а), нужные команде.
2. Проверить доступы к проекту и инструментам (read/write, git, shell).
3. Связать MCP с используемой IDE/клиентом агента.
4. Протестировать 2-3 реальных сценария: поиск по коду, правка, git-операции.

Ответ на вопрос "работает ли без IDE":

- MCP сам по себе может работать вне конкретной IDE (зависит от клиента/агента).
- Но на практике самый удобный и продуктивный сценарий для разработки - через IDE (Cursor/VS Code), где есть контекст файлов, diff и ревью в одном месте.

## 3) EDT + IDE (Cursor / VS Code)

### 3.1 Выбор решения / модели

Критерии выбора:

- **Точность в коде 1С и рефакторинге** - приоритет для сложной логики.
- **Скорость ответа** - важна для рутинных правок.
- **Цена/лимиты** - для массового использования в команде.
- **Стабильность на длинном контексте** - для крупных задач.

Практика:

- Для быстрых итераций - "быстрая" модель.
- Для сложной генерации/рефакторинга - более "сильная" модель.
- Для критичных изменений - всегда ручной ревью и тесты.

### 3.2 Настройка плагинов, MCP, правил (rules)

Минимальная конфигурация:

1. Установить IDE и плагины работы с кодом/репозиторием.
2. Подключить MCP-конфигурацию.
3. Добавить project rules: стиль, запреты, политика изменений.
4. Описать repeatable-команды (например: создание PR, формат проверок, проверка diff).

Принципы:

- Не перегружать правилами "на будущее".
- Добавлять rules по мере повторяющихся ошибок.
- Команды автоматизировать только после того, как процесс стабилен.

### 3.3 Методология промптов в Cursor/IDE

Лучший рабочий шаблон промпта:

1. **Цель**: что нужно получить в итоге.
2. **Область**: какие файлы/модули можно менять.
3. **Ограничения**: что нельзя менять.
4. **Критерии готовности**: как понять, что задача выполнена.
5. **Проверка**: какие тесты/сценарии прогнать.

Практики из best practices:

- Начинать с плана для нетривиальных задач.
- Давать только релевантный контекст, без "мусора".
- Если знаете файл - указывать точный путь.
- При смене задачи открывать новую сессию, чтобы не тащить старый шум.
- Для исторического контекста использовать ссылки на прошлые чаты, а не копипаст огромных кусков.

### 3.4 Фичи Cursor/IDE по работе с кодом

Полезные возможности:

- inline-редактирование и точечная генерация;
- режим агента для многошаговых задач;
- работа с diff в процессе генерации;
- быстрый переход по проекту и поиск контекста;
- интеграция с git-процессами (коммит, PR-сценарии, ревью).

Рекомендация по горячим клавишам:

- Зафиксируйте в команде единый "минимум" шорткатов: вызов агента, принятие/отмена автодополнения, переход к diff, запуск поиска.
- Добавьте внутренний cheatsheet под вашу ОС (Windows/macOS/Linux), так как раскладки отличаются.

## 4) Кейсы

### 4.1 Разработка внешнего отчета

1. Сформулировать структуру отчета и источники данных.
2. Создать объект и каркас в EDT.
3. Сгенерировать типовые участки кода через ИИ (запросы, обработка параметров, вывод).
4. Проверить производительность и корректность на реальных данных.
5. Зафиксировать результат в отдельной ветке и пройти ревью.

### 4.2 Разработка внешней обработки

1. Описать сценарий использования и входные параметры.
2. Сделать минимально рабочую версию.
3. Добавить ИИ-генерацию только для повторяющихся участков.
4. Проверить права, безопасность и поведение на ошибках.

### 4.3 Доработка типового функционала

- Предпочитать расширения вместо правок "ядра", где это возможно.
- Если правка типовой неизбежна, изолировать изменения и документировать причину.
- Перед обновлениями иметь карту затронутых объектов.

### 4.4 Доработка функционала расширения

- Работать через отдельные ветки под задачу.
- Избегать скрытых зависимостей на нестабильные объекты основной конфигурации.
- После изменений проверять совместимость с актуальной версией базы.

### 4.5 Работа с обновлением баз

#### 4.5.1 Если обновилась основная база - что делать

1. Обновить локальную рабочую базу до той же версии.
2. Синхронизировать репозиторий с актуальной веткой.
3. Пересобрать проект в EDT, устранить конфликты/несовместимости.
4. Прогнать регрессионные сценарии по затронутым модулям.

#### 4.5.2 Можно ли обновлять базу через EDT? Особенности

- Да, но процесс должен быть стандартизирован в команде.
- Ключевые риски: несовместимость версий, конфликт расширений, разъезд метаданных.
- Обновления делать только по чек-листу и желательно в отдельном техническом окне/ветке.

## 5) Работа с Git

### 5.1 Подключение проекта к Git с нуля

1. Инициализировать репозиторий (`git init`) или подключить существующий.
2. Настроить `.gitignore` под EDT/служебные файлы.
3. Сделать первый коммит текущего корректного состояния.
4. Подключить удаленный репозиторий и отправить ветку.

### 5.2 Подключение разработчика к проекту с Git

1. Клонировать репозиторий.
2. Проверить доступы и веточную модель.
3. Импортировать проект в EDT и убедиться, что он собирается.
4. Создать личную feature-ветку и начать работу.

### 5.3 Инструкция разработчика по Git (ежедневный цикл)

1. `pull` актуальной базовой ветки.
2. Работа в feature-ветке маленькими логическими коммитами.
3. Самопроверка diff перед коммитом.
4. PR + ревью + исправления.
5. Слияние только после зеленых проверок.

### 5.4 Git + EDT: как работать с ветками и ничего не сломать

#### 5.4.1 Выгрузка изменений

- Выгружать только свою задачу, без "чужого шума".
- Перед коммитом убедиться, что нет случайных массовых переформатирований.
- Коммитить атомарно: один смысл - один коммит.

#### 5.4.2 Загрузка изменений

- Сначала подтянуть изменения базовой ветки.
- Разрешить конфликты локально и пересобрать проект.
- Проверить ключевые сценарии перед пушем.

#### 5.4.3 Merge

- Не мержить "вслепую": обязательно просматривать diff.
- В конфликтных местах приоритет у рабочей бизнес-логики, а не у механического выбора "ours/theirs".
- После merge - smoke/regression-тесты.

### 5.5 Поведение при обновлении базы через N-релизов

Рекомендованный порядок:

1. Зафиксировать текущую рабочую точку (ветка + теги/релизные метки).
2. Последовательно пройти промежуточные релизы (где это критично).
3. На каждом шаге синхронизировать EDT-проект и устранять конфликты.
4. После финального обновления прогнать полный набор критичных сценариев.
5. Только после стабилизации продолжать функциональную разработку.

## Рекомендуемый командный стандарт (кратко)

- Планировать нетривиальные задачи до генерации кода.
- Давать ИИ ограниченный, но достаточный контекст.
- Работать малыми изменениями и регулярно ревьюить diff.
- Не отключать человеческий контроль: тесты и код-ревью обязательны.
- Поддерживать единые версии инструментов и единый git-процесс по команде.
