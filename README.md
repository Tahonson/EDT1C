## 1. Быстрый старт (если вы раньше работали только в Конфигураторе)

Цель: за 1 проход поднять рабочую среду и начать доработки через EDT + Git + Cursor.

### 1.1 Что поставить (минимум)

1. Платформа 1С: `8.3.27+` ([releases.1c.ru](https://releases.1c.ru/)).
2. EDT: ветка `2026.x` ([releases.1c.ru](https://releases.1c.ru/)).
3. Java: JDK, совместимый с выбранной EDT ([bell-sw.com/downloads](https://bell-sw.com/pages/downloads/)).
4. Cursor + расширения для 1С ([cursor.com](https://www.cursor.com/)).

### 1.2 Линейный старт

1. Сделайте резервную копию базы.
2. Закройте Конфигуратор с этой базой.
3. В EDT выполните импорт:
   - `Файл -> Импорт -> 1C:Enterprise -> Конфигурация из информационной базы`.
4. Укажите пользователя с полными правами и корректную версию проекта.
5. Если есть расширения — отметьте их при импорте.
6. Дождитесь построения/индексации EDT.
7. Подключите Git (через EDT UI; терминал — альтернативно).
8. Привяжите ИБ в свойствах проекта EDT и выполните `F7`.
9. Для внешних обработок/отчетов создайте отдельный проект EDT и работайте через него.
10. В Cursor установите 1С-расширения и проверьте поиск по `*.bsl`.

### 1.3 Состав рабочей области EDT (обязательный минимум)

Рекомендуемая структура одной рабочей области EDT для одного прикладного решения:

- один проект основной конфигурации;
- один или несколько проектов расширений;
- один проект внешних отчетов/обработок;
- одна привязанная ИБ для запуска и отладки.

Это снижает рассинхрон между исходниками, внешними `.epf/.erf` и тем, что реально запускается в отладке.

Ссылочная база для продолжения: раздел `7. Карта полезных материалов и ссылок`.

## 2. EDT + Напарник

<a id="sec-1-1"></a>
### 1.1 Требования и развертывание

Алгоритм установки рабочей среды:

1. Платформа 1С. Скачайте с [releases.1c.ru](https://releases.1c.ru/) и установите платформу `8.3.27+`.
2. EDT. Скачайте с [releases.1c.ru](https://releases.1c.ru/) актуальную версию ветки `2026.x`.
3. Java. Установите `BellSoft Liberica JDK` (версия — согласно требованиям скачанной EDT).
4. Плагин Напарника. В EDT откройте `Help -> Install New Software`, добавьте репозиторий плагина Напарника и установите плагин.
5. Авторизация. Перезапустите EDT, затем в появившемся окне выполните авторизацию под УЗ портала ИТС.

<a id="sec-1-3"></a>
### 1.3 Генерация кода по промпту

<details>
<summary>Видео-нарезка: 1С:Напарник (генерация по промпту)</summary>
<a href="https://www.youtube.com/watch?v=V441Ih3T3NE&t=2271s"><img src="https://img.youtube.com/vi/V441Ih3T3NE/hqdefault.jpg" alt="Видео: 1С:Напарник (генерация по промпту)" width="900" /></a>
</details>

Алгоритм генерации:

1. Установите курсор в пустую строку внутри процедуры.
2. Напишите комментарий-инструкцию (промпт).
Пример хорошего промпта:

```bsl
// Сформируй запрос к регистру накопления ОстаткиТоваров. Обороты, условие по Складу (из параметров), сгруппируй по Номенклатуре.
```
Пример плохого промпта:

```bsl
// Получи остатки
```
3. Нажмите `Enter`. Напарник сгенерирует текст ниже.
4. Обязательная проверка: убедитесь, что имена реквизитов/переменных совпадают с реальными метаданными.

<a id="sec-1-4"></a>
### 1.4 Правила комментирования

1. Обычные комментарии (`//`): используйте для постановки задач Напарнику внутри алгоритма.
2. Документирующие комментарии: строго перед объявлением метода.

Алгоритм:

1. Введите `//` перед процедурой и нажмите `Enter`. EDT создаст шаблон.
2. Заполните типы и назначение параметров. Это критично для контекста AI в других модулях.

Пример:

```bsl
// Получает артикул по ссылке на номенклатуру.
// Параметры:
//  Товар - СправочникСсылка.Номенклатура - Искомый элемент.
// Возвращаемое значение:
//  Строка - Артикул товара.
```

## 3. EDT + MCP

<a id="sec-2-1"></a>
### 2.1 Настройка MCP для 1С (для работы в Cursor)

Алгоритм подключения:

1. Убедитесь, что MCP-сервер для 1С доступен в вашей среде.
2. В Cursor откройте настройки: `Cursor Settings -> Features -> MCP`.
3. Добавьте MCP-сервер и проверьте, что есть нужные вам права (read/write там, где это разрешено).
4. Протестируйте 2–3 реальных сценария: поиск по коду, разбор структуры проекта, правки и git-операции (в рамках разрешений).

**Скриншот (замените): настройки MCP в Cursor:**

![Скриншот: настройки MCP (замените)](docs/screens/shot-cursor-mcp-settings.png)

Шаблон `mcp.json`:

```json
{ "mcpServers": { "1c-docs": { "url": "https://bulzavr.ru/mcp" }, "EDT MCP Server": { "type": "sse", "url": "http://localhost:8765/mcp" } } }
```

На практике самый удобный сценарий — работать через Cursor/VS Code: там проще держать контекст файлов и видеть diff рядом с изменениями.

## 4. EDT + IDE (Cursor / VS Code)

<a id="sec-3-1"></a>
### 3.1 Выбор решения / модели
Настройки моделей (Cursor -> Settings -> Models):

1. `Primary model` (для сложных задач): `Claude, Gemeni, Chat GPT` — для архитектуры и сложных запросов/ задач.
2. `Fast model` (для мелких правок): `можно авто и остальные модели, которые уже есть` — для быстрых итераций. Не рекомендуемый вариант.

Для критичных изменений всегда делайте ручную проверку и прогон сценариев.

<a id="sec-3-2"></a>
### 3.2 Настройка плагинов, MCP, правил (rules)

<details>
<summary>Видео-нарезка: Cursor (MCP/IDE)</summary>
<a href="https://www.youtube.com/watch?v=V441Ih3T3NE&t=2562s"><img src="https://img.youtube.com/vi/V441Ih3T3NE/hqdefault.jpg" alt="Видео: Cursor (MCP/IDE)" width="900" /></a>
</details>

Алгоритм настройки рабочей области:

1. Установите расширения для 1С в Cursor:
   - поддержка BSL (подсветка/навигация),
   - дерево конфигурации 1С,
   - CodeTour (для маршрутов по проекту и онбординга).
2. Быстрая установка расширений (через `Ctrl+P` -> `ext install ...`):
   - `ext install 1c-syntax.language-1c-bsl`
   - `ext install whiterabbit.1c-configuration-tree`
2. Откройте `Cursor Settings -> Features -> MCP`.
3. Подключите минимум 2 MCP:
   - `1c-docs` (документация и справочный контекст),
   - `EDT MCP Server` (контекст проекта EDT).
4. Создайте файл `.cursorrules` в корне проекта.
5. Вставьте базовые правила (пример):

```text
Язык программирования: 1С (BSL).
Использовать только русский синтаксис.

Политика изменений:
- Менять только то, что требуется задачей (без “скрытых” архитектурных рефакторингов).
- Перед любыми изменениями, влияющими на метаданные, сверять имена реквизитов/методов по текущему проекту (через EDT).
- По возможности расширять функциональность через расширения, а не через правку “ядра”.
```

- дополнительные MCP под задачи команды (кроме `1c-docs` и `EDT MCP Server`);

<a id="sec-3-2-1"></a>
### 3.2.1 Горячие клавиши VS Code/Cursor (минимум)

Быстрое управление:

- `Ctrl+Shift+P` или `F1` — палитра команд;
- `Ctrl+P` — быстрый переход к файлу;
- `Ctrl+,` — настройки;
- `Ctrl+B` — показать/скрыть боковую панель;
- ``Ctrl+` `` — показать/скрыть встроенный терминал.

Навигация и правка:

- `F12` / `Alt+F12` / `Shift+F12` — переход, предпросмотр и ссылки;
- `F2` — переименование;
- `Ctrl+G` — переход к строке;
- `Ctrl+/` — комментарий строки;
- `Shift+Alt+F` — форматирование документа;
- `Ctrl+.` — быстрые исправления.

<a id="sec-3-3"></a>
### 3.3 Методология промптов в Cursor/IDE
Алгоритм постановки задачи (Ctrl+L / Cmd+L):

1. Дайте контекст через `@`.
Пример: `@Документ.Реализация.МодульОбъекта.bsl @ОбщийМодуль.Скидки.bsl`
2. Опишите задачу структурно.
Пример: В прикрепленном модуле документа найди процедуру `ОбработкаПроведения`. Добавь вызов функции `РассчитатьСкидку` из модуля `Скидки` перед записью движений.
3. Вложения: если документ/форма незнакомы — добавьте через `@` XML файл структуры (например `Document.Реализация.xml`), чтобы Cursor “увидел” реквизиты.
4. Критерии готовности: что должно получиться и как проверить.

<a id="sec-3-4"></a>
### 3.4 Фичи Cursor/IDE по работе с кодом

Полезные возможности:

- inline-редактирование и точечная генерация;
- режим агента для многошаговых задач;
- работа с diff в процессе генерации;
- быстрый переход по проекту и поиск контекста;
- интеграция с git-процессами (коммит, PR-сценарии, ревью).

Рекомендация по горячим клавишам:

- Зафиксируйте в команде единый "минимум" шорткатов: вызов агента, принятие/отмена автодополнения, переход к diff, запуск поиска.
- Добавьте внутренний cheatsheet под вашу ОС (Windows/macOS/Linux), так как раскладки отличаются.

<a id="sec-3-5"></a>
### 3.5 Практика работы в Cursor (по обучающему видео)

Видео: [Как работать с Cursor](https://www.youtube.com/watch?v=abZzOaA2ZFc&source_ve_path=OTY3MTQ&embeds_referring_euri=https%3A%2F%2Fbulzavr.ru%2F)

Базовый рабочий цикл:

1. Открывайте проект через Explorer и используйте быстрый поиск:
   - `Ctrl+P` — поиск файла по имени;
   - `Ctrl+Shift+E` — фокус на проводник;
   - `Ctrl+Shift+X` — расширения.
2. Установите два обязательных расширения для 1С:
   - дерево конфигурации 1С;
   - поддержка языка BSL (подсветка/навигация).
3. Для больших конфигураций открывайте нужную подсистему/конфигурацию отдельно (меньше шума в поиске и навигации).
4. Для перехода к коду модулей используйте поиск по `*.bsl` и дерево конфигурации; формы и визуальные редакторы удобно оставлять в EDT.

Практика по AI-режиму в Cursor:

1. Для первичного анализа задачи используйте `Auto` и короткий контекстный промпт.
2. Если ответ неточный — уточняйте промпт и добавляйте релевантный контекст (`@`-файлы, путь, скриншот ошибки).
3. Не перегружайте запрос нерелевантной информацией: это ухудшает точность.
4. Сгенерированные запросы/SQL/BSL проверяйте вручную: модель может усложнить запрос без необходимости.

<a id="sec-4"></a>
## 5. Кейсы

<a id="sec-4-1"></a>
### 4.1 Доработка основной конфигурации (типовой код)

Алгоритм в EDT:

1. Откройте объект/модуль в EDT.
2. Внесите правку вручную или через Напарника в EDT.
3. Проверьте изменения, обновите ИБ (`F7`) и воспроизведите сценарий.

Алгоритм в Cursor:

1. Найдите нужный `*.bsl` через поиск/дерево конфигурации 1С.
2. Передайте задачу в Chat с контекстом `@` (затронутые модули и ограничения).
3. Примените изменения в файле и вернитесь в EDT.

## 6. Работа с Git

<a id="sec-5-1"></a>
### 5.1 Подключение проекта к Git с нуля

<details>
<summary>Видео-нарезка: Git в EDT</summary>
<a href="https://www.youtube.com/watch?v=V441Ih3T3NE&t=814s"><img src="https://img.youtube.com/vi/V441Ih3T3NE/hqdefault.jpg" alt="Видео: Git в EDT" width="900" /></a>
</details>

Основной сценарий (через EDT UI):

1. Откройте Git-перспективу/панели Git в EDT.
2. Инициализируйте или привяжите репозиторий к текущему проекту EDT.
3. Проверьте список изменений в staging-панели EDT.
4. Выполните первый commit и отправку в удаленный репозиторий из EDT.

Альтернативный сценарий (терминал):

1. Выгрузить конфигурацию в проект EDT (папка `src`).
2. Открыть терминал в корневой папке проекта.
3. Выполнить `git init`, создать `.gitignore`, сделать `git add .` и первый commit.
4. Подключить remote и отправить основную ветку.

<a id="sec-5-2"></a>
### 5.2 Подключение разработчика к проекту с Git

Алгоритм начала работы:

1. Клонировать репозиторий (замените URL на ваш):
   - `git clone URL_репозитория` (замените на ваш URL репозитория)
2. Импортировать проект в EDT:
   - `Файл -> Импорт -> General -> Проекты из папки или архива`
3. Привязать информационную базу для запуска (ПКМ по проекту -> `Свойства -> 1С:Предприятие`):
   - создать пустую ИБ и привязать её
4. Выполнить `F7` и убедиться, что сборка/обновление проходит.

<a id="sec-5-3"></a>
### 5.3 Инструкция разработчика по Git (ежедневный цикл)

Алгоритм выполнения задачи:

1. Создайте feature-ветку (через EDT UI; терминал — как альтернатива).
2. Пишите код в EDT/Cursor только в рамках своей задачи.
3. Подготовка к commit:
   - в EDT откройте Git Staging/Синхронизация,
   - проверьте, что изменены только нужные объекты/файлы.

**Скриншот (замените): Git Staging/Синхронизация в EDT:**

![Скриншот: Git Staging (замените)](docs/screens/shot-git-staging.png)

4. Сделайте commit (из EDT UI; терминал — альтернативно).
5. Отправьте ветку в удаленный репозиторий.
6. Создайте PR/MR, пройдите ревью, внесите правки, затем merge.

<a id="sec-5-4"></a>
### 5.4 Взаимодействие Git и EDT (Выгрузка/Загрузка)

При получении изменений (`git pull`):

1. Выполните `git pull` в терминале.
2. В EDT дождитесь перестроения рабочей области (фоновый процесс).
3. Выполните `F7` (применение изменений к локальной базе 1С).

При смене веток (`git checkout`):

1. Смените ветку (в Cursor/терминале).
2. В EDT на корне проекта выполните `F5` и дождитесь перестроения.
3. Выполните `F7`.

При мерджах:

- Не мержьте “вслепую”: обязательно просматривайте diff и понимайте влияние на бизнес-сценарии.
- После merge прогоните воспроизводимые smoke/regression-тесты.

Параллельная работа команды (минимум):

1. Одна задача — одна feature-ветка.
2. В начале рабочего цикла синхронизируйтесь с целевой веткой команды.
3. Перед merge: проверка diff + обновление ИБ + smoke-сценарий.
4. Конфликты разбирайте на уровне бизнес-логики, а не механически.

<a id="sec-5-5"></a>
### 5.5 Алгоритм обновления базы (Слияние с веткой поставщика)

<details>
<summary>Видео-нарезка: обновление базы (в конфигураторе)</summary>
<a href="https://www.youtube.com/watch?v=V441Ih3T3NE&t=2010s"><img src="https://img.youtube.com/vi/V441Ih3T3NE/hqdefault.jpg" alt="Видео: обновление базы (в конфигураторе)" width="900" /></a>
</details>

Самый критичный процесс. Строго соблюдайте последовательность.

Исходные данные: у вас есть ветка `vendor` (чистая конфигурация поставщика) и ветка `master/develop` (ваша доработка).

Алгоритм обновления:

1. Обновление ветки `vendor`:
   - `git checkout vendor`
   - обновите базу в EDT (F5)
   - в EDT: ПКМ на проекте -> `Импорт` -> `1С:Предприятие` -> `Конфигурация из файла`
   - выберите новый `.cf` от 1С и дождитесь окончания разбора
   - `git add .` и `git commit -m "vendor: Обновление типовой конфигурации до релиза X.X.X.X"`
2. Подготовка к слиянию:
   - `git checkout master` (или вашу ветку разработки)
   - в EDT дождитесь перестроения проекта на старую ветку
3. Запуск слияния:
   - в терминале: `git merge vendor`
4. Разрешение конфликтов (Merge Tool):
   - если конфликтов нет — переходите к шагу 6
   - если конфликты есть — в EDT откройте Merge Tool по конфликтному метаданному объекту
   - перенесите нужные блоки кода и реквизиты в центральную панель, сохраните
   - в терминале завершите merge после разрешения конфликтов (при необходимости отдельным commit)

**Скриншот (замените): конфликты метаданных в дереве EDT:**

![Скриншот: конфликты (замените)](docs/screens/shot-merge-conflicts-tree.png)

**Скриншот (замените): Merge Tool (3 панели):**

![Скриншот: Merge Tool (замените)](docs/screens/shot-merge-tool-window.png)
5. Завершение merge-коммита:
   - `git commit -m "merge: Слияние релиза X.X.X.X с доработками"`
6. Обновление базы:
   - в EDT выполните `F7` и проведите стандартное обновление базы данных 1С (при необходимости через F5/запуск предприятия для обработчиков обновления).

## Рекомендуемый командный стандарт (кратко)

- Планировать нетривиальные задачи до генерации кода.
- Давать ИИ ограниченный, но достаточный контекст.
- Работать малыми изменениями и регулярно ревьюить diff.
- Не отключать человеческий контроль: тесты и код-ревью обязательны.
- Поддерживать единые версии инструментов и единый git-процесс по команде.
